<objective>
Close the REQ-046 residual tracker-staleness gap (Phase 09 deferred-items Finding 1): a server-resolved combat turn that produces EMPTY narration persists no new master message, so the client safety-poll's "masterCount > baseline" never trips and the combat tracker (turn pointer + PC/monster HP) can stay stale even though turn_advance / hp_change already changed state — especially when the completion SSE event is dropped.

The combat events themselves already fire a `state` refresh when dispatched with sessionId (emitStateRefresh in tools.ts:200), but that SSE can be DROPPED (the exact scenario the safety-poll exists for), and the empty-narration turn gives the poll no new-message signal to recover on. The turn-level empty-narration branch is INLINE in route.ts (~739-754: `if (_finalNarration.trim()) { persist master message } else { notifySession turn-error }`), NOT in the vault loop module.

Locked discretion (10-CONTEXT.md): pick the SMALLEST correct fix — a server-side `state` NOTIFY on the empty-narration combat turn, or a client recovery that refetches the snapshot independent of a new master message. Because REQ-046 explicitly includes "when the completion SSE event is dropped," the durable recovery (the polled/fetched recovery path) is the load-bearing half; a NOTIFY alone can be dropped the same way. Default to: (a) a server `state` notify on the empty-narration combat turn as the fast path, plus (b) confirming/extending the client recovery so the snapshot refetch is not gated solely on a new master message. The executor may implement only ONE if they justify it closes BOTH the empty-narration AND dropped-SSE cases.

Purpose: the tracker reflects the current encounter within one turn with no manual page refresh, even on a dropped SSE + empty-narration turn.
Output: the chosen fix + a regression test.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server → SSE/poll → client tracker | The combat-state refresh signal crosses to the browser; the signal must be durable against a dropped SSE event. |
| turn route → notifySession | A pg_notify payload is capped at 8000 bytes (notify.ts:49); the `state` payload is tiny (no data), well within bounds. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-03-DOS | Denial of Service | refetch storm from an over-eager recovery / repeated state notify | mitigate | Guard the extra state notify on combatStateChanged + empty-narration only (one notify per turn); debounce any client recovery refetch to one call per recovery tick; do not loop. |
| T-10-03-INFO | Information Disclosure | the `state` notify payload | accept | The `{type:'state'}` payload carries NO data — it only tells the client to refetch the authenticated GET /api/sessions/:id snapshot, which enforces its own authz. No PII crosses in the notify. |
| T-10-03-REGR | Tampering (regression) | non-combat / message-persisting turns | mitigate | The fix is gated on combatStateChanged + empty narration (server) or guarded on turn-completion recovery (client); regression suite + full-suite delta confirm unchanged behavior elsewhere; REQ-022 prompt hash untouched (no prompt-builder edit). |
| T-10-03-DROP | Denial of Service (availability) | a dropped completion SSE leaving the tracker stale | mitigate | The durable half (client snapshot refetch on the poll/finalizedSeq recovery, not gated on a new master message) closes the dropped-SSE case independent of any single SSE delivery. |
