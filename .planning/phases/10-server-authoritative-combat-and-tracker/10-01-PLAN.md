<objective>
Create the pure, deterministic, headless-testable encounter OPENER helper (option B) that PRODUCES the ordered list of encounter events the existing v1/v2 resolver stack already consumes. This is the core of REQ-045: the server — not the local model — decides combat is starting and emits combat_start + monster_spawn[] + initiative_set.

This plan is helper-only. NO route wiring (that is 10-02). NO edits to resolveCombat / runMonsterTurnLoop / the CR→stats table / getBestiaryAttackStats / resolveCombatHandoff — the opener only feeds them.

Purpose: a pure function is the verification surface (the route has no unit test). Mirroring the shipped pure `resolveCombat` (combat-resolver.ts) and `resolveMonsterTurn`/`runMonsterTurnLoop` (monster-turns.ts) — injectable RNG, events out, never throws — keeps the opener deterministic in tests and swappable later (option A drops into the monster-selection step).
Output: `open-encounter.ts` (pure helper) + a seeded headless unit suite.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| player free-text → monster selection | The playerMessage is untrusted free text; the opener parses an enemy name + count from it. |
| opener → events.md (via the route in 10-02) | The opener only PRODUCES event objects; persistence/path handling is the route's dispatchVaultTool (10-02). This helper performs no fs access. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01-ReDoS | Denial of Service | the enemy-name/count parse regex in open-encounter.ts | mitigate | Use bounded regexes only — anchored character classes, no nested quantifiers, no `(.*)+` style alternation; mirror the bounded patterns in monster-bestiary.ts (ATTACK_HIT_RE/DAMAGE_DICE_RE, lines 35/40). Run the parse on the message ONCE, not per-block-greedy. |
| T-10-01-SPAM | Denial of Service | unbounded monster spawn from "attacco mille goblin" | mitigate | Cap parsed count at MAX_SPAWN (named const) and `console.warn` the cap; the helper never emits more than MAX_SPAWN monster_spawn events. |
| T-10-01-PATH | Tampering | monster id / name derivation from player text | mitigate | This helper does NO file access — it never builds a path. Name→bestiary path lookup happens in 10-02 through the existing path-safe `readVaultFile`/`safeVaultPath` (getBestiaryAttackStats), never string-concatenated here. Monster ids are server-generated (`m-<slug>-<index>`), not raw player text used as a path. |
| T-10-01-THROW | Denial of Service | malformed input crashing the turn | mitigate | Never throws — a parse failure falls back to one generic CR-default monster; asserted by Task 1 case 9/10. |
</threat_model>
