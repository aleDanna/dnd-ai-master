<objective>
Add the missing SRD statblock reader that REQ-045 needs to spawn monsters with their real seeded HP/AC/CR. Today `getBestiaryAttackStats(name)` returns only attack prose ({attackBonus, damageDice}); there is NO reader that returns {hpMax, ac, cr}. Without it, the opener's bestiaryLookup always returns null and EVERY monster — goblin and dragon alike — gets a generic CR-default HP, silently degrading REQ-045 ("select monster stats from the seeded SRD bestiary") to "CR-guess for everything."

This plan adds a NEW exported async function `getBestiaryStatblock(name) → { hpMax?: number; ac?: number; cr?: string } | null`, colocated in monster-bestiary.ts, that reads `handbook/monsters/<slug>.md` through the same `readVaultFile` helper `getBestiaryAttackStats` already uses (readVaultFile routes through `safeVaultPath` — path-traversal/ReDoS-safe — NEVER string-concat a path), then parses the leading `---`-delimited YAML frontmatter block IN THE NEW FUNCTION to read the `hpMax`, `ac`, and `cr` fields. IMPORTANT: the monster stats live in FRONTMATTER, not in the `## Actions` prose that getBestiaryAttackStats parses, and there is NO existing YAML/frontmatter parser in monster-bestiary.ts (or anywhere in src/) — the new function does a small inline parse of the frontmatter block. The frontmatter key is `hpMax` (verified against the committed files — read it directly, no rename); `ac`/`cr` are read directly too. Adding a new exported reader is explicitly allowed by the v1/v2-untouched constraint; the existing `getBestiaryAttackStats` math/prose-parse is NOT edited.

This is the second Wave-1 plan (parallel with 10-01). It plugs into the opener as the injected `bestiaryLookup` (10-01) and is wired into production by the route (10-03). Splitting it out also resolves the "10-02 was too heavy" warning by giving the bestiary-stats path its own focused TDD plan.

Purpose: Make REQ-045 real — the goblin spawns with hpMax 7 from the SRD, not a CR guess.
Output: NEW `getBestiaryStatblock` export + its test file.
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|

No string concatenation of raw input into a path. |
| T-10-05 | Denial of Service | malformed/huge frontmatter | mitigate | The inline frontmatter parse reads only the bounded leading `---` block and coerces only hpMax/ac/cr; return null on read/parse failure or missing block rather than throwing, so a bad file cannot crash the turn route. |
| T-10-06 | Information disclosure | reading arbitrary files | mitigate | readVaultFile/safeVaultPath confine reads to the vault handbook tree; traversal inputs return an `ERROR` marker → null. |
