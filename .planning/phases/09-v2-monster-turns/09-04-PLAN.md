# Plan 09-04 — Monster-turn loop driver + combined Italian narration

<plan_metadata>
**Wave:** 2 | **Depends on:** 09-01, 09-02, 09-03 | **Autonomous:** yes
</plan_metadata>

<objective>
Build the **monster-turn loop driver** (`runMonsterTurnLoop`) that resolves consecutive
monster turns by composing the Wave-1 primitives: `resolveMonsterTurn` (09-02, pure roll +
target + hit rule), `getBestiaryAttackStats` (09-03, Level-1 stats), and the existing
`applyEncounterEvent` projector reducer (in-memory turn/state advance). The core loop
logic — stop conditions, safety cap, target filtering — is **pure and headless-testable**
with an injected RNG; the **only** I/O is the isolated Level-1 bestiary lookup, which
null-returns cleanly on failure so the loop never aborts (falls through to the CR table /
defaults). After the loop, build ONE combined Italian 2nd-person narration directive
listing every monster action's outcome (D-15).
</objective>

<deviations_from_pattern>
None — this plan follows the v1 resolver + narration-directive pattern established in Phase 08.
</deviations_from_pattern>

<tasks>

<title>Loop driver: runMonsterTurnLoop with stop conditions, safety cap, party-KO</title>
<read_first>

- .planning/phases/09-v2-monster-turns/09-RESEARCH.md (Pattern 6 loop driver, Pattern 8 combined narration, Pitfalls 1/3/6)
</read_first>
<action>
Implement `runMonsterTurnLoop` in `src/app/api/sessions/[id]/turn/monster-turns.ts`. The loop
repeatedly: reads the active actor from the working `EncounterState` (`turnOrder[currentIdx]`),
stops if it is a live PC / no live targetable PC remains / the `MONSTER_LOOP_SAFETY_CAP` is hit,
otherwise calls `resolveMonsterTurn` for the active monster, collects the returned events, applies
them to an in-memory working copy via `applyEncounterEvent` (+ manual PC-HP decrement for hp_change),
and repeats. Signature: `runMonsterTurnLoop({ encounter, pcAcById, pcHpById, rng? })` returning
`{ events, narrationFacts, stoppedReason }`. Track each turn's outcome (monster name, hit/miss,
damage, target PC) in `narrationFacts` for the directive builder.
</action>
<acceptance_criteria>
- monster-turns.ts exports `runMonsterTurnLoop`
- `npx vitest run tests/app/api/sessions/[id]/turn/monster-turns.test.ts` exits 0
- Loop stops on all three conditions (live PC active / no live target / safety cap) — asserted with seeded RNG
- Last live PC at 0 HP → loop stops, stoppedReason indicates party KO, no combat_end event emitted
- A null from getBestiaryAttackStats falls through to CR/default stats (loop does not abort)
</acceptance_criteria>
</task>
