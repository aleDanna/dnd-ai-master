<objective>
Implement the monster-turn loop driver and the combined-narration builder in `src/app/api/sessions/[id]/turn/monster-turns.ts`, composing the pure primitives from 09-02, the bestiary parse from 09-03, and the `cr` from 09-01.

The driver's roll/target/stop/damage CORE is pure and headless-testable: it operates on an in-memory `EncounterState` working copy and a `Map<pcId, hpCurrent>` working copy, applying each emitted event via `applyEncounterEvent` so it never re-reads disk, and `resolveMonsterTurn`/`getMonsterAttackStats` are pure with an injectable RNG (the route in 09-06 owns persistence). The ONE async, filesystem-touching piece is the Level-1 bestiary lookup `getBestiaryAttackStats` (09-03) — `runMonsterTurnLoop` is `async` only because it `await`s that call per living monster. That bestiary read is isolated behind the 3-level fallback: 09-03 returns `null` on any miss/error (it never throws), and `getMonsterAttackStats` treats a `null` bestiary result as "use the CR table (Level 2) / default (Level 3)". So a bestiary read failure mid-loop is absorbed cleanly — the monster simply uses CR/default stats and the loop continues; it never aborts. Per iteration: find the active actor (`turnOrder[currentIdx]`); if it is a live monster, resolve its 3-level attack profile (bestiary → cr-table → default), call `resolveMonsterTurn` over the LIVE PC ids (HP>0 in the working copy), accumulate the result, apply the emitted events to the working state + decrement the target PC's working HP (clamped at 0). Advance via the emitted `turn_advance` (already in the result events).

Stop conditions (D-03 + D-14): (a) the active actor is a live PC; (b) no live targetable PC remains (last-PC-KO) → stop + party-KO signal; (c) `MONSTER_LOOP_SAFETY_CAP` reached → clean stop. NEVER throw (v1 D-10 "edge → resolve defensively"). After the loop, build ONE combined narration directive (D-15) listing every outcome.

Purpose: this is the orchestration the route hooks. It must be headless-testable (RESEARCH Pattern 6) so the operator smoke and unit tests assert deterministic loop behavior. Keeping the core pure-in-memory (apply events to a working state rather than re-reading events.md per iteration) also sidesteps the RESEARCH Pitfall 6 N-reads concern; the only I/O it performs is the isolated, defensively-null-returning bestiary read.
Output: `runMonsterTurnLoop` returning accumulated results + stopReason + one combined narration directive.
</objective>
