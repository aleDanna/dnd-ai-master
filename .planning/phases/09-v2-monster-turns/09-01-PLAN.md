<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| monster_spawn payload → validateEvent | the `cr` value enters the system via the event log; validateEvent is the input-validation gate (ASVS V5) |
| validated event → EncounterState | `cr` flows into state only via the server-controlled reducer; player input never sets it |
| EncounterState.cr → CR table (downstream) | a malformed cr surviving into state would crash the table — but validateEvent already bounds it; the table (09-02) also defaults defensively |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Input Validation / DoS | `cr` in monster_spawn payload | mitigate | validateEvent rejects non-number / NaN / Infinity / negative cr with an explicit error before it ever reaches state — a malformed cr cannot crash the downstream CR-table lookup (ASVS V5, RESEARCH Security Domain "malformed cr → DoS"). |
| T-09-02 | Tampering | event replay / schema change | mitigate | Change is strictly additive (optional field on both type and state); byte-stable replay tests prove cr-less logs project identically — no migration, no replay divergence (D-08 back-compat). |
| T-09-03 | Tampering | player-influenced monster stats | mitigate | `cr` is copied verbatim from the server-controlled `monster_spawn` event only; the reducer introduces no new player-input surface. Monster mechanics stay server-authoritative (ASVS V4). |
</threat_model>
