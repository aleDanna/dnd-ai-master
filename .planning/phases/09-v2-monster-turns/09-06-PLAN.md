2. **Gate + loop (D-01/D-02):** read the post-v1 encounter (the same replay), and if `vaultMutationsEnabled && encounter.active` and `turnOrder[currentIdx].actorId` is a live monster, call `runMonsterTurnLoop({ encounter, pcAcById, pcHpById })`. 3. **Narration (D-15/D-16) — CORRECTNESS-CRITICAL, PRIMARY FLOW per D-01:** set a `_monsterLoopRan` flag; pass `monsterResolved: _monsterLoopRan` to `buildTurnDirective`; extend the `suppressCombatMutations` gate to `(_resolver !== null || _monsterLoopRan)`; inject the loop's combined `narrationDirective` via `appendDirectiveToHistory`. **Bind the FINAL narration via `enforceResolvedNarration` whenever `_monsterLoopRan` is true — regardless of `_resolver` state.** Per D-01 the monster loop fires right after the player's message is processed and any resulting `turn_advance` (including the v1 player-attack resolution), so the COMMON path is: player attacks on their turn → v1 resolver runs (`_resolver !== null`) → the turn advances to a monster → the loop runs (`_monsterLoopRan`) — BOTH true in the SAME request. If the guard stayed `_resolver === null` (player-only), that common path would inject the monster directive but leave the final text UNbound, leaking the model's competing roll-asks / event-JSON. The monster path passes a `ResolveCombatResult`-shaped object with `damageRequest: null` (the monster sequence has no single settled damage-request); the combined monster directive (D-15) is what governs that final, sanitized pass.
4. **Handoff preserved:** the existing post-loop `resolveCombatHandoff` (07-03) still runs and hands to the next PC.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client turn request → route | the player message is untrusted; the monster loop is gated on server-derived EncounterState (active actor is a monster), never on player-supplied monster ids |
| campaignId → dispatchVaultTool | campaignId is server-authoritative (`campaign.id`), never player-derived (mirrors T-08-06) |
| PC AC/HP (Postgres + vault) → loop | PC defense values come from server state; the player cannot redirect or inflate them |
| loop events → events.md | emission goes through the validating dispatcher; the loop runs OUTSIDE the DB transaction to avoid long-held-tx / deadlock |
| LLM narration → client | on a loop-ran turn the model can leak competing roll-asks / event-JSON; the final text must be sanitized before it reaches the player |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-09-19 | DoS | loop inside HTTP request | mitigate | The loop is bounded by MONSTER_LOOP_SAFETY_CAP (09-04) and runs OUTSIDE the DB transaction (RESEARCH Anti-Pattern) so a degenerate loop cannot hold a transaction or hang within the M4 latency budget; the whole block is in try/catch and never breaks the turn. |
