## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client turn request → route | the player message is untrusted; the monster loop is gated on server-derived EncounterState (active actor is a monster), never on player-supplied monster ids |
| campaignId → dispatchVaultTool | campaignId is server-authoritative (`campaign.id`), never player-derived (mirrors T-08-06) |
| PC AC/HP (Postgres + vault) → loop | PC defense values come from server state; the player cannot redirect or inflate them |
| loop events → events.md | emission goes through the validating dispatcher; the loop runs OUTSIDE the DB transaction to avoid long-held-tx / deadlock |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-18 | Tampering | monster id / target selection | mitigate | The loop reads the active monster + targets from server-resolved EncounterState + the party PC-id set; the player never supplies a monster id or a target. campaignId is always `campaign.id` (server-authoritative) for every dispatchVaultTool call (ASVS V4, mirrors T-08-06). |
| T-09-19 | DoS | loop inside HTTP request | mitigate | The loop is bounded by MONSTER_LOOP_SAFETY_CAP (09-04) and runs OUTSIDE the DB transaction (RESEARCH Anti-Pattern) so a degenerate loop cannot hold a transaction or hang within the M4 latency budget; the whole block is in try/catch and never breaks the turn. |
| T-09-20 | Tampering | double-apply (LLM re-emits loop events) | mitigate | suppressCombatMutations is extended to the loop turn (drops the LLM's combat apply_event calls) AND buildTurnDirective receives monsterResolved (no combat re-ask); enforceResolvedNarration strips any leaked event-JSON/roll-asks (RESEARCH Pitfall 3, mirrors Phase-08 T-08-04). |
| T-09-21 | DoS | hp_change underflow / huge delta | mitigate | The loop clamps its working HP (09-04) and the existing hp_change reducer clamps `max(0, hp+delta)` — a hostile/large damage bottoms PC HP at 0, never negative (RESEARCH Security Domain, D-13). |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase (RESEARCH: pure in-repo extension, no new dependencies) — supply-chain surface unchanged; no [ASSUMED]/[SUS] packages to gate. |
